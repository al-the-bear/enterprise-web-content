import 'package:tom_flutter_ui/src/forms/list/tom_list_form_toolbar.dart';

class TomListFormToolbar

class TomListFormToolbar<TItem extends InvalidType, TForm extends TomForm<TItem>> extends TomContainerBase

Compact action bar with the standard list-form mutations: **Add**, **Remove**, **Move up**, **Move down**.

Drop into any list-rendering widget (master pane, inline list footer, expandable header) so each demo / app doesn't reinvent the same row of buttons. Each action is enabled only when it is meaningful for the current selection (e.g. *Move up* disabled at index 0; *Remove* disabled when nothing is selected).

**Layout.** Buttons are rendered inside a single [Wrap]. The reorder pair (Move up / Move down) is bundled into an inner [Row] so the `Wrap` treats it as one atomic child — the two arrows always stay next to each other, and they only flow onto a new line *as a pair* when the host doesn't have enough horizontal space for them to fit alongside the wide tonal-filled Add/Remove buttons. This avoids the lopsided break (one arrow next to Add/Remove, the other arrow alone on the second line) reported against narrow master panes, while still letting the toolbar lay out inline when there's room.

Example:

TomListFormToolbar(
  listForm: form.shippingAddresses,
  selectedIndex: selected,
  onSelectionChanged: selectIndex,
)

Selection wiring: * After **Add**, selection moves to the newly appended row. * After **Remove**, selection clears. * After **Move up / down**, selection follows the moved row.

Supplying [onSelectionChanged] is optional — if omitted, selection changes are still applied to the list but not reported back. That mode is useful when the host already observes [TomListForm] and derives selection some other way.

Pass `showXxx: false` to hide individual buttons, or override [beforeActions] / [afterActions] to inject custom buttons before / after the standard set.

**Resource resolution.** The toolbar follows the standard Tom resource pattern. Each button's label / tooltip / icon falls back to resources when the matching constructor argument is null:

**Authorization.** The whole toolbar respects the standard [TomPolicyNodeBase] gate at `<basePath>` (none → toolbar disappears, disabled → all buttons greyed out, read → readonly treatment, full → normal). In addition, each button consults its own sub-resource path (`<basePath>.add`, `<basePath>.remove`, `<basePath>.moveUp`, `<basePath>.moveDown`) — when its sub-state is `none` the button is hidden regardless of `showXxx`; when `disabled` or `read` it is rendered without an `onPressed` callback so the user cannot fire it.

Per-row authorization is not supported by design — the same toolbar auth applies to every row of the list. Authorize individual fields of the row sub-form via the field widgets themselves.
