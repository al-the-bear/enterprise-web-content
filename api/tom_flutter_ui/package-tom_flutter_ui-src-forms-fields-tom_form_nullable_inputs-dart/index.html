Phase 4 — Nullable input variants.

Mirror the surface of the non-nullable Phase 1 / Phase 3 inputs but bind a T? value so the user can express "explicitly empty" as a distinct state from "default / pristine". Each variant ships with a trailing clear icon that sets the value to null; for text inputs, clearing the text also yields null.

Concretes: [TomFormNStringInput] — TomField<String?> text input. [TomFormNIntInput] — TomField<int?> numeric input. [TomFormNDoubleInput] — TomField<double?> decimal input. [TomFormNDateInput] — TomField<DateTime?> date picker button.

final TomStringInputDecorationMixin<dynamic> decorationField

Same instance as [field], but typed so we can call the decoration mixin methods (applyErrorVisualsTo, decorateField, resolveStyle) and the _NullableFieldLabels getters without an is check that the analyzer can't promote across the generic boundary.

final String Function() nullDisplay

Resolves the placeholder shown in place of the value when the bound field is null. Closure so each concrete can wire it through its own _NullableFieldLabels mixin instance (the static analyzer can't see the mixin across the generic boundary).

typedef _NullableInputDecorator

typedef _NullableInputDecorator = TomStringInputDecorationMixin<dynamic>

Type alias for whatever-the-concrete-field is (one of the three textual nullable variants). Holds the methods needed to lay out the input — applyErrorVisualsTo, decorateField, resolveStyle come from the TomStringInputDecorationMixin; labelText/hintText/ resolveNullDisplay() from the _NullableFieldLabels mixin.
