void bind(T target)

Bind to a TomClass instance.

target.getValues() carries the whole state subtree (including every nested TomClass), and TomClass.setValues recurses through that subtree writing values into matching member observables. On top of that this method walks every nested [TomForm] (via [nestedForms]) and runs the binding lifecycle — initial snapshot, per-field _initialValue reset, pristine validation state, and FieldBoundEvent / FormBoundEvent dispatch — all the way down.

void _attachBindingObserver(InvalidType target)

Round-6 (a): subscribe to the bound TomClass so external mutations (e.g. a sibling [TomForm] bound to the same target committing an edit) re-snapshot this form's fields. Without this, two scaffolds targeting the same data object only stay in sync for the side that initiated the edit.

void _detachBindingObserver()
void _onBindingExternallyChanged()

Fires when the bound [TomClass] mutates from outside this form — typically because a sibling form (e.g. a master-detail scaffold's standalone detail form) wrote new values to the same TomClass, or because a nested list inside the binding had elements added/removed through a sibling [TomListForm].

Always fires [_changes] so any UI reading aggregate state (notably [isDirty] for save/discard buttons) re-renders. The dirty flag itself can flip from external mutations — e.g. a sibling form removes an element from a nested TomList, which changes our nested [TomListForm.hasStructuralChanges] from false to true. Without firing here, the outer form's discard button would stay disabled.

Snapshot-restore is gated below and only runs when the form is not dirty: - if dirty, leave field values alone so we don't clobber in-flight user input; - if not dirty, re-snapshot from the (now externally-updated) binding so the UI shows the latest data.

Re-entrancy: skipped entirely while we are writing back to the binding ourselves ([_writingBackToBinding] guard for [_applySave]).

void _applyBindLifecycle(InvalidType target)

Core of [bind]: per-form lifecycle reset + recursion into nested forms. Assumes values have already been written (via [setValues] on the root form), so the recursion itself never sets values — it only commits each level's "current values are the new initial".

Future<bool> save({bool? commitBaselines})

Save form values back to bound state.

Runs full validation (including nested forms' AND child containers' validations) first — returns false if any errors. On success, commits every field in the form tree, snapshots each child container, and fires FormSavedEvent at each level.

Nested-save semantics. When the user calls save() on a form that has an outer ancestor in the form tree (see [hasFormAncestor]), the save is treated as an in-memory commit only: values are written back to the bound business object, but no field, snapshot, or list-form _initialOrder is updated. That keeps the outer ancestor's transactional baseline intact so the user can still discard the whole transaction from above. Conversely, a save() initiated on the outermost (root) form cascades a full baseline commit through the entire subtree — matching the user's intent that "this is the new saved state".

[commitBaselines] is passed by internal cascade calls and should normally be left at its default for user code — the public entry point auto-detects the right policy from [hasFormAncestor].

void _applySave({required bool commitBaselines})
void discard()

Discard edits — resets every field in this form, every nested form, and every child container (e.g. TomListForm) back to its last bound / committed value, then writes the initial snapshot back to the binding.

The write-back is necessary because sibling forms (e.g. a master-detail scaffold's detail pane) may have written to the same TomClass between bind and discard. Without the write-back, the backing state would retain those external edits even after the outer form says "Discard".

TomList members are intentionally excluded from the write-back because _initialSnapshot stores raw TomClass instances for list fields (getValues() returns the live list, not a serialized copy), so setValues() cannot deserialise them. Instead, each TomListForm child container restores its list structure via TomList.resetWith(_initialOrder) in its own discard() above, which fires TomListLifecycleListener hooks to atomically dispose the old sub-forms (onBeforeClear) and recreate them (onAfterReset).

void unbind()

Detach from the bound state object.

Resets all fields (including nested form fields) to pristine, unbinds every nested form and child container, then clears the binding reference and the initial snapshot at every level. Call this when the form tree is no longer editing a data object.

Future<List<FormValidationError>> validate()

Validate every field in this form, every nested form, and every child container (e.g. TomListForm); then run this form's cross-field validation. Returns the combined error list and fires a single FormValidatedEvent carrying all of it — child containers also fire their own events through the bus, but downstream TomErrorViews subscribed to the parent see one merged snapshot.

class TomTextFormFieldBase

class TomTextFormFieldBase<T> extends TomField<T>

Base class for text-based form fields with TextEditingController support.

Individual [InputDecoration] parameters (e.g. [labelText], [helperText], [prefixIcon]) follow the resource lookup pattern: 1. Explicit parameter value (highest priority) 2. Resource lookup via scope path ($scope.$tomId.$suffix) 3. null / absent (lowest priority)

This allows resources to provide default values while code can override any specific field at construction time.
