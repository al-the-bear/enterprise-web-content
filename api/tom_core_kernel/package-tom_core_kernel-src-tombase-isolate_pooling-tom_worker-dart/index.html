Isolate pooling and worker management for concurrent task execution.

This library provides a robust framework for managing isolates in Dart/Flutter applications. It enables parallel processing by spawning worker isolates that can execute commands independently from the main isolate.

Key Features

  • Worker Isolates: Spawn and manage isolated execution contexts
  • Bidirectional Communication: Send commands to workers and receive responses
  • Command Pattern: Execute commands as objects with built-in result handling
  • Executor Pattern: Register handlers to process incoming messages
  • Worker Pools: Manage multiple workers with automatic load distribution

Example Usage

// Create a worker context
final context = TomWorkerContext(
  TomEnvironment.development,
  TomPlatform.mobile,
  [],
  'MyWorker',
);

// Spawn a worker
final worker = await TomWorker.spawn(
  TomWorker.startRemoteIsolate,
  context,
);

// Execute a command
final result = await worker.executeCommand(myCommand);

// Clean up
worker.close();

See also: - [TomWorker] for individual worker management - [TomWorkerPool] for managing multiple workers - [TomExecutor] for custom command handlers - [TomCommand] for encapsulated command objects

class TomCommand

abstract class TomCommand

Base class for self-executing commands sent to worker isolates.

Unlike executors which are registered handlers, commands encapsulate both data and behavior. When a command is received by a worker, its [execute] method is called automatically.

When to Use

Use [TomCommand] when: - The command logic is self-contained - You want to send data and functionality together - Higher isolation between command types is desired

Use [TomExecutor] when: - Multiple command types share common handling logic - You need a central point for command processing - Commands are simple data objects

Example

class ComputeFactorialCommand extends TomCommand {
  final int number;
  ComputeFactorialCommand(this.number);

  @override
  Future<Object?> execute() async {
    return factorial(number);
  }
}

// Send to worker
final result = await worker.executeCommand(ComputeFactorialCommand(10));

class TomExecutor

abstract class TomExecutor

Base class for command executors that handle messages in worker isolates.

Executors provide a way to define custom handlers for incoming commands. Multiple executors can be registered, and they are called in order until one produces a result (i.e., returns something other than [noResult]).

Pattern Overview

The executor pattern allows separation of command handling logic from the worker infrastructure. Each executor can focus on a specific type of command or domain.

Example

class MyExecutor extends TomExecutor {
  @override
  Object execute(Object command) {
    if (command is MyCommand) {
      return processMyCommand(command);
    }
    return TomExecutor.noResult; // Let next executor handle it
  }
}

// Register the executor
TomWorker.executors.add(MyExecutor());

Important Notes

  • Return [noResult] to indicate the command was not handled
  • Executors exist in both isolates and may handle different commands
  • The creator-side worker handles responses from the created worker

Constructors

TomExecutor.new()

Methods

Object execute(Object command)

Executes a command and returns the result.

This method is called when a command is received by the worker. The [command] parameter contains the data sent to the isolate.

Returns: - The result of processing the command - [noResult] if this executor cannot handle the command

Note: This method is called in both creator and worker isolates. The TomWorker exists in both isolates and both sides can have different workers with different executor sets.

Static

static Object noResult

Sentinel value indicating that an executor did not process a command.

Return this value from [execute] to allow the next executor in the chain to handle the command.

class TomWorker

class TomWorker

Manages communication with a worker isolate.

[TomWorker] provides the core functionality for spawning isolates and managing bidirectional communication between the main isolate and worker isolates.

Architecture

┌─────────────────┐         ┌─────────────────┐
│  Main Isolate   │◄───────►│ Worker Isolate  │
│   (Creator)     │ SendPort│   (TomWorker)   │
│                 │◄───────►│                 │
│  TomWorker      │ReceiveP │  Executors      │
│  instance       │         │                 │
└─────────────────┘         └─────────────────┘

Example

// Spawn a worker
final worker = await TomWorker.spawn(
  TomWorker.startRemoteIsolate,
  TomWorkerContext(env, platform, [], 'Worker'),
);

// Execute commands
final result = await worker.executeCommand(MyCommand());

// Check if busy
if (worker.isBusy()) {
  print('Worker is processing requests');
}

// Shutdown
worker.close();

Initializes a worker isolate and starts the message handling loop.

This method should be passed as the initializer to [spawn]. It sets up the communication channels and calls the context's initialization hooks.

  • [context]: The worker context containing configuration

Throws [TomWorkerException] if the context's sendPort is null.

  TomWorker.startRemoteIsolate, // Use this method
  context,
);
static bool canSendToCreator()

Returns true if backchannel communication to the creator is available.

This returns true only when called from within a worker isolate. In the main isolate, this always returns false.

static void sendToCreator(Object o)

Sends a message to the creator isolate (fire-and-forget).

Use this for one-way communication from a worker to the main isolate. For request-response patterns, use [executeInCreator] instead.

  • [o]: The object to send to the creator

Throws [TomWorkerException] if called from the main isolate.

Note: Do not add logging in this method to avoid stack overflow when using separate isolates for logging.

static Future<Object?> executeInCreator(Object command)

Executes a command in the creator isolate and awaits the result.

Use this for request-response communication from a worker to the main isolate. The command will be processed by executors registered on the creator's [TomWorker] instance.

  • [command]: The command to execute in the creator

Returns the result of executing the command.

Throws [TomWorkerException] if called from the main isolate.

Example:

// From within a worker isolate:
final config = await TomWorker.executeInCreator(GetConfigCommand());

class TomWorkerContext

class TomWorkerContext

Configuration context for initializing worker isolates.

This class holds the configuration and state needed to spawn and initialize a worker isolate. It provides hooks for custom initialization logic in both the isolate and worker contexts.

Lifecycle

1. Context is created with environment configuration 2. [initializeIsolate] is called in the new isolate (static context) 3. [initializeWorker] is called on the TomWorker instance

Example

class MyWorkerContext extends TomWorkerContext {
  MyWorkerContext() : super(
    TomEnvironment.development,
    TomPlatform.mobile,
    [],
    'MyWorker',
  );

  @override
  Future<bool> initializeIsolate() async {
    // Set up isolate-specific resources
    await setupDatabase();
    return true;
  }

  @override
  Future<bool> initializeWorker(TomWorker worker) async {
    // Configure the worker instance
    worker.sendExecutorToIsolate(MyExecutor());
    return true;
  }
}

class TomWorkerException

class TomWorkerException extends TomException

Exception thrown when worker-related operations fail.

This exception is thrown in scenarios such as: - Attempting to start an isolate without a configured SendPort - Trying to send messages to creator from the creator isolate itself - General worker communication failures

Example:

try {
  TomWorker.sendToCreator(message);
} on TomWorkerException catch (e) {
  print('Worker error: ${e.defaultUserMessage}');
}

class TomWorkerPool

class TomWorkerPool

Manages a pool of worker isolates for parallel task execution.

[TomWorkerPool] provides automatic load balancing by distributing commands across multiple workers. When all workers are busy, requests are queued until a worker becomes available.

Use Cases

  • CPU-intensive computations that benefit from parallelization
  • Batch processing of independent tasks
  • Request handling where isolation is required

Example

// Create a pool with 4 workers
final pool = await TomWorkerPool.withSize(4, context);

// Execute commands - automatically distributed
final results = await Future.wait([
  pool.execute(ProcessCommand(data1)),
  pool.execute(ProcessCommand(data2)),
  pool.execute(ProcessCommand(data3)),
  pool.execute(ProcessCommand(data4)),
]);

Notes

  • Workers are created during pool initialization
  • Commands wait if all workers are busy (polling every 1ms)
  • Consider pool size based on CPU cores and task characteristics

Methods

Object? execute(Object command)

Executes a command using an available worker from the pool.

If all workers are busy, this method waits (polling every 1ms) until a worker becomes available. The command is then executed and the worker is returned to the pool.

  • [command]: The command to execute

Returns the result of executing the command.

Example:

final result = await pool.execute(MyCommand());

Static

static Future<TomWorkerPool> withSize(int size, TomWorkerContext context)

Creates a worker pool with the specified number of workers.

Each worker is spawned using [TomWorker.spawn] with the provided context. The method waits for all workers to be ready before returning.

  • [size]: Number of workers to create
  • [context]: Configuration for each worker

Returns a [Future] that completes when all workers are ready.
