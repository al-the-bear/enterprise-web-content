Tom Authorization — UI authorization primitives for the Tom framework.

This file defines the core authorization types used by the UI layer: - [TomAuthState] — the four-state authorization outcome - [TomAuthorizable] — interface for elements that participate in authorization - [TomAuthorizationGroup] — group of related authorizables with collective control - [TomAuthorization] — global singleton for resolving authorization state - [TomAuthorizer] — per-widget/action authorizer that can modify auth outcomes - [TomAuthorizerStrategy] — how authorizer interacts with resource authorization - [TomUIStateController] — build-time callback for narrowing widget state

class TomAuthorizable

abstract interface class TomAuthorizable

Interface for elements that participate in the Tom authorization system.

Implementers gain: - [tomId] for resource-based authorization lookups - [tomGroup] for group-level visibility control - [effectiveAuthState] — the resolved authorization state for this element

Implemented by: - Tom widgets (via TomNodeBase, TomWidgetMixin) - Tom actions (via TomAction) - Menu item definitions, named routes/screens

class TomAuthorization

class TomAuthorization

Global authorization singleton.

Initialized at app startup with a concrete adapter. Widgets, forms, and actions access it directly — no InheritedWidget or BuildContext needed. 

// App startup
TomAuthorization.adapter = MyAuthorizationAdapter();

// Usage anywhere
final auth = TomAuthorization.resolveAuth('checkout.email');

class TomAuthorizationGroup

class TomAuthorizationGroup

A group of related Tom elements that share authorization control.

Members implement [TomAuthorizable], ensuring type-safe access to [tomId] and [authorizationGroup] properties. 

final shippingGroup = TomAuthorizationGroup('shipping',
    mode: AuthGroupMode.controlsAll);
TomTextField(tomId: 'street', authorizationGroup: shippingGroup, ...),
TomTextField(tomId: 'city', authorizationGroup: shippingGroup, ...),

Group paths follow the pattern: `<scopePath>.<groupId>`.

class TomAuthorizer

class TomAuthorizer

Per-element authorizer that can modify the authorization outcome.

Attached to individual widgets or actions. The authorizer is informed of the resource-based authorization state and can modify it according to the [TomAuthorizerStrategy]. 

TomElevatedButton(
  tomId: 'save',
  authorizer: TomAuthorizer(
    strategy: TomAuthorizerStrategy.authorizerNarrowsOnly,
    resolve: (resourceAuth) {
      // Disable save when no changes exist
      if (!hasChanges) return TomAuthState.disabled;
      return resourceAuth;
    },
  ),
)

class TomUIStateController

class TomUIStateController

Build-time callback that can further narrow a widget's effective state.

The main difference between authorization and UI state control: - **Authorization** happens at widget creation time (construction-time) - **UI state controller** runs at build time and can further narrow based on dynamic business rules

The controller CANNOT widen authorization — it can only narrow.

TomElevatedButton(
  tomId: 'reorder',
  uiStateController: TomUIStateController(
    resolve: (currentAuth, data) {
      // Disable reorder when list is empty
      if (items.isEmpty) return TomAuthState.disabled;
      // Check row-specific ownership
      if (data case {'ownerId': int id} when id != currentUserId) {
        return TomAuthState.read;
      }
      return currentAuth;
    },
  ),
)

Constructors

Resolves the final UI state given the current authorization state.

The [currentAuth] is the combined result of resource authorization and any [TomAuthorizer]. The return value can only narrow this state.

The optional [data] parameter provides context for row-level decisions, such as checking if the current user owns a list item.

enum AuthGroupMode

enum AuthGroupMode

Determines how a [TomAuthorizationGroup]'s auth state applies to its members.

Values

controlsAll

Group state replaces all member states (after form-level narrowing).

controlsWidest

Group state is the ceiling — members can be narrower but not wider.

controlsNarrowest

Group state is the floor — members that would be more restrictive are elevated to the group's state.

enum TomAuthState

enum TomAuthState

The four authorization states for a UI element.

These map to distinct visual/behavioral outcomes: - [none] — element is hidden (as if it doesn't exist) - [disabled] — element is visible but greyed out, no content shown; indicates data may exist but cannot be seen or might not exist - [read] — element shows current value/state in read-only mode - [full] — element is fully interactive/editable

Values

none

Hidden — the element does not exist from the user's perspective.

disabled

Disabled — visible but greyed out, no content. Indicates data may exist but cannot be accessed or might not exist for this object.

read

Read-only — displays current value/state but cannot be modified.

full

Full access — the element is fully interactive and editable.

enum TomAuthorizerStrategy

enum TomAuthorizerStrategy

Strategy for how a [TomAuthorizer] interacts with resource authorization.

  • [authorizerDecides]: The authorizer has full control. It receives the

resource authorization state and can return any state, including widening. - [authorizerNarrowsOnly]: The authorizer can only narrow (further restrict) the resource authorization. It cannot widen access.

Values

authorizerDecides

The authorizer has full control over the final state. It receives the resource auth state and can return any state.

authorizerNarrowsOnly

The authorizer can only narrow the resource authorization. The final state is the more restrictive of resource auth and authorizer result.
