User identity, principal, and access control information types.

This library provides the core data structures for representing authenticated users and their access rights in the Tom framework.

Key Concepts

  • **User**: Basic identity information (name, email, etc.)
  • **Principal**: Authenticated session with full context
  • **ACI**: Access Control Information (roles, groups, entitlements)
  • **CLI**: Client Limits Information (quotas, enabled features)

Authorization Model

The framework supports flexible authorization:

1. **Roles**: Job functions (admin, manager, user) 2. **Groups**: Organizational units (can be nested) 3. **Entitlements**: Fine-grained permissions 4. **Resource Keys**: Field-level access control

Usage Example

// Check user's roles
var principal = TomBearerAuthentication.getPrincipal();
if (principal.aci.roles.contains('admin')) {
  showAdminPanel();
}

// Access user information
print('Hello, ${principal.user.firstname}');
print('Organization: ${principal.organization}');

See also: - [TomAccessControl] for access control annotations - [TomBearerAuthentication] for token management

class TomAccessControlDefinition

class TomAccessControlDefinition

Global ACI schema definition for resolving complete permissions.

This class defines the authorization model for the application, specifying how roles, groups, and entitlements relate to each other and to resource keys.

Inheritance Model

  • **Roles** can include other roles, grant group memberships,

entitlements, and resource keys - **Groups** can include roles, other groups, entitlements, and resource keys - **Entitlements** can include other entitlements and resource keys

Resolution Process

When [completeFromDefinition] is called with a user's basic ACI, it recursively resolves all inherited permissions to produce the complete set of effective permissions.

Example

var definition = TomAccessControlDefinition(
  roles: [
    TomRole(
      name: 'admin',
      entitlements: ['admin.*'],
      resourceKeys: ['*'],
    ),
    TomRole(
      name: 'user',
      entitlements: ['feature.read'],
    ),
  ],
  groups: [
    TomGroup(
      name: 'engineering',
      rolesIncluded: ['user'],
    ),
  ],
);

// Resolve complete permissions for a user
var completeAci = definition.completeFromDefinition(userAci);

class TomAccessControlInformation

class TomAccessControlInformation

Groups together authorization information for a user.

The exact meaning of "roles", "groups", "entitlements" and "resourceKeys" depends on the authorization concept of your application.

Authorization Concepts

  • **Roles**: Job functions or responsibilities (e.g., 'admin', 'user')
  • **Groups**: Organizational units; can be nested with inheritance
  • **Entitlements**: Fine-grained permissions for features
  • **Resource Keys**: Field-level access control identifiers

Simplified Authorization

For simple applications, you can use just one approach:

1. Public vs. authenticated users only 2. Role-based access control (RBAC) 3. Group-based access control 4. Entitlement-based access control 5. Resource key-based access control

Mixing approaches is supported but not encouraged.

Example

var aci = TomAccessControlInformation(
  roles: ['admin', 'user'],
  groups: ['engineering'],
  entitlements: ['feature.reports.view'],
  resourceKeys: ['user.email', 'user.phone'],
);

if (aci.roles.contains('admin')) {
  // Grant admin access
}

During login, these fields must be populated from your authorization system. The definitions may change over time through administration.

class TomClientLimitsInformation

class TomClientLimitsInformation

Controls system capabilities independent of user rights.

This class enforces subscription-based limits and feature flags. Even if a user has the rights to access a feature, it may be disabled or quota-limited based on the client's subscription.

Use Cases

  • Feature flags for gradual rollouts
  • Subscription tier limits (e.g., API call quotas)
  • Module enablement per client

Example

var cli = TomClientLimitsInformation(
  enabledModules: ['core', 'reporting'],
  quotas: {'api_calls': 1000, 'storage_mb': 500},
);

if (cli.enabledModules.contains('reporting')) {
  // Show reporting features
}

if (currentUsage < cli.quotas['api_calls']!) {
  // Allow API call
}

class TomEntitlement

class TomEntitlement

Represents an entitlement in the authorization model.

Entitlements are fine-grained permissions that can be assigned to users, groups, or roles. They can include other entitlements and grant access to specific resource keys.

Naming Convention

Entitlements typically follow a hierarchical naming pattern: - `feature.action` (e.g., `reports.view`) - `module.feature.action` (e.g., `billing.invoices.create`)

Example

var readReports = TomEntitlement(
  name: 'reports.read',
  resourceKeys: ['report.data', 'report.metadata'],
);

var manageReports = TomEntitlement(
  name: 'reports.manage',
  entitlements: ['reports.read', 'reports.create', 'reports.delete'],
);

class TomGroup

class TomGroup

Represents a group in the authorization model.

Groups organize users and can inherit permissions from other groups and roles. They're typically used to represent organizational units, teams, or departments.

Inheritance

Groups can: - Include roles (members get those roles) - Be members of other groups (nested groups) - Have direct entitlements - Have direct resource keys

Example

var engineeringGroup = TomGroup(
  name: 'engineering',
  rolesIncluded: ['developer'],
  groupMemberShips: ['all-employees'],
  entitlements: ['repo.access'],
);

class TomPrincipal

class TomPrincipal

Represents an authenticated user session with full context.

The principal combines user identity, authorization information, and session context into a single object. It's the primary way to access the current user's information throughout the application.

Components

  • **User**: Identity information ([user] property)
  • **ACI**: Access control information ([aci] property)
  • **CLI**: Client limits information ([cli] property)
  • **Context**: Session-specific data ([currentContext])

Two-Factor Authentication

If [requires2FA] is true, the user has authenticated with username/password but still needs to complete 2FA. The [twoFactorType] indicates which method to use.

Localization

The principal tracks both application and client locale settings: - [language], [country], [timezone]: Application/server settings - [clientLanguage], [clientCountry], [clientTimezone]: Client device settings

Example

var principal = TomBearerAuthentication.getPrincipal();

// Check authentication
if (principal.authenticated) {
  print('Hello, ${principal.user.firstname}!');
}

// Check authorization
if (principal.aci.roles.contains('admin')) {
  showAdminPanel();
}

// Add session context
principal.addToContext({'lastActivity': DateTime.now()});

class TomRole

class TomRole

Represents a role in the authorization model.

Roles define job functions or responsibilities. They're the primary mechanism for role-based access control (RBAC).

Inheritance

Roles can: - Include other roles (role hierarchies) - Grant group memberships - Have direct entitlements - Have direct resource keys

Example

var adminRole = TomRole(
  name: 'admin',
  rolesIncluded: ['user'],  // admins get all user permissions
  entitlements: ['admin.*'],
  resourceKeys: ['*'],
);

var userRole = TomRole(
  name: 'user',
  entitlements: ['feature.read', 'profile.edit'],
);

class TomUser

class TomUser

Represents a user's identity information.

Contains basic profile information like name, email, and contact details. This is the identity portion of the authentication, separate from the authorization information in [TomAccessControlInformation].

Guest Users

Unauthenticated or anonymous users are represented with the special [guestUsername] value.

Privacy Protection

User fields can be protected using resource keys. When serialized to tokens or transmitted, protected fields may be hidden based on the recipient's access rights.

Example

var user = TomUser(
  id: 123,
  username: 'jdoe',
  firstname: 'John',
  lastname: 'Doe',
  email: 'john.doe@example.com',
  timeZone: 'America/New_York',
);

// Check if guest
if (user.username == TomUser.guestUsername) {
  promptLogin();
}
